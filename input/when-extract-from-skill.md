# 什么时候该把计算从 Skill 里拆出来

让 AI 计算贝叶斯算法只有 20% 的准确率

你以为 AI 在算数，其实它在玩成语接龙.

就拿我设计的「贝叶斯 Skill」来说，让它预测明天会下雨么。每次给它一点新内容：闻到了泥土味，看到了石头出汗、墙壁返潮，家养的金鱼把头浮出了水面，看到了蚂蚁正搬家。它每次都会预估一个概率，然后依照贝叶斯算法给出后面的概率。

| 阶段              |   H₁ 明显下雨   | H₂ 多云不雨 | H₃ 短暂飘雨 |  H₄ 晴朗  |
| --------------- | :---------: | :-----: | :-----: | :-----: |
| **🟦 先验（随机打分）** |   **25%**   | **30%** | **15%** | **30%** |
| ① 预估｜闻到泥土味      |     65%     |   45%   |   55%   |   15%   |
| ① **计算后验**      | **38.2%** ↑ | 31.8% ↑ | 19.4% ↑ | 10.6% ↓ |
| ② 预估｜石头出汗       |     75%     |   40%   |   65%   |    8%   |
| ② **计算后验**      | **52.3%** ↑ | 23.2% ↓ | 23.0% ↑ |  1.5% ↓ |
| ③ 预估｜蚂蚁搬家       |     55%     |   35%   |   45%   |   15%   |
| ③ **计算后验**      | **60.6%** ↑ | 17.1% ↓ | 21.8% ↓ |  0.5% ↓ |
| ④ 预估｜金魚浮头       |     50%     |   38%   |   45%   |   30%   |
| ④ **计算后验（最終）**  | **64.8%** ↑ | 13.9% ↓ | 21.0% ↓ |  0.3% ✗ |

我刚开始的时候偷懒，让 AI 自己计算概率。它算出来分数看上去像那么回事，我也没深究。直到有一天我给 Skill 加上了测试，才发现问题。

论文《Probabilistic Reasoning in Generative LLMs》提到，让 LLM 计算贝叶斯只是在「猜」数字。正确率只有 20%，哪怕是有「思维链」COT 的推导模式，也只有 65% 的正确率。而换成脚本计算的正确率可以高达 97%。

我让 Claude Code 把计算拆出来，它忙了一会儿，脚本已经写好了。

有些日常的 Skill 会做确定的工作。比如：刚才贝叶斯例子里的公式计算， 从 PDF 里提取文字，从一个网页里抽出所有的图片链接，把 Markdown 转成 HTML 文件。这类东西有个特点：能找到现成算法，或者固定输入输出、每次跑出来结果都一样。遇到这种，直接让 Claude Code 拆成脚本，再让 Skill 调用就行。

如果你不清楚这里有没有「确定性」的内容，你也可以把这篇文章丢给 AI，让它自己找。你要是能碰到那种「即确定又反复使用」的内容，你就挖到宝了。原来跑一次要消耗几百 token、等两三秒的计算，脚本接管之后几乎是瞬间出结果，而且 token 一分不多花。Anthropic 官方的 Skill 创建提示词里也是这么建议的。

如果你只是盯着那种反复使用的功能拆，你可能会漏掉另一种「脆弱操作」的情况。假设你正在一条长长的吊桥上，两边是深不见底的悬崖。你不敢往下看，只能用双手抓紧两旁又粗又扎手的麻绳，慢慢地往前挪步。

日常工作里有很多这种「脆弱操作」的情况，比如修改数据库结构之后的迁移操作，一步踩错了就掉到深渊里了。你需要按照固定的先后顺序操作，而通常也只有这么一条正确的路。如果让 AI 随着自己的性子自由发挥，你后面根本没办法用 Ctrl+Z 来补救。

还有一类是「让脚本来管理进度」的情况，Skill 同样没有办法记住一套流程的正确顺序，它可能会跳步，不按照你要求的顺序执行或自己添加一些不存在的步骤。这次先不展开，如果你想了解，写信告诉我，我单独写一篇。

遇到一个 Skill 不确定该不该拆，我的判断顺序是：有没有确定答案？会不会出错？出错了能不能撤回？三个问题，任何一个答「是」，就值得拆。

不过，有些东西不该拆。比如「根据用户的语气决定回复的风格」，或者「判断这段话的意思是什么」，这种事情没有固定答案，也找不到现成算法，拆成脚本反而什么都做不了。规则是：有确定答案的交给脚本，没有确定答案的留给 LLM。

当然也不能无止境地多拆。另一篇论文《Less is More：Optimizing function call for LLM Execution on Edge Devices》分析了为什么过多的脚本调用会更难为 LLM。你让它在一堆脚本里找合适的调用，它很可能想了半天，最后选了错的给你。减少拆出来的脚本数量，执行时间最多能缩短 70%。你把这个原则告诉 AI，让它来决定拆哪些就行。

罗马不是一天造成的。Skill 也不是一上来就要先拆脚本。实际的做法是「没问题就不动它」。等真实的问题出现了，再升级，这是防止过度设计最管用的办法。比如你先让 Claude Code 设计一个 Skill，通常 Claude Code 会默认把所有内容都写入一个 SKILL.md 文件。后面你发现每次跑出来的结果都不一样，今天对、明天错，再升级到拆脚本。拆出来的脚本最好是 Python 3 + 内置库，因为每台 Mac 电脑都预装了这些环境。你从自己的 Mac Book Air 换到家里另一台的 Mac Mini，复制过来的 Skill 也能正常使用。

回想一个你常用的 Skill，是不是有过上面提到的那些情况，如果是，就看看能不能拆出脚本来。
